# AI IDE 部署过程与问题复盘

## 1. 文档目的

本文档用于记录本次 AI IDE 在阿里云 ECS 上的部署过程、过程中出现的问题、已经形成的约束，以及后续推荐的部署方案，便于后续继续接手部署、排障和复盘。

说明：

- 本文档基于本次实际处理过程整理，不是理想化 SOP。
- 敏感信息（服务器密码、API Key 等）不在本文档中明文保留。
- 当前已形成明确约束：`super-admin` 保持使用 `80` 端口，AI IDE 使用 `8080` 端口，不再占用或改动 `80` 端口上的现有服务。

## 2. 部署背景

### 2.1 服务与环境

- 服务器：阿里云 ECS
- 系统资源：`1G` 内存
- 部署方式：Docker
- 后端：FastAPI + Python
- 前端：Vue 3 + Vite
- 数据库：PostgreSQL 16 + pgvector

### 2.2 端口约束

- `super-admin`：占用 `80`
- AI IDE 后端：容器内 `8000`
- AI IDE 前端：对外 `8080`

### 2.3 关键硬约束

- 不能破坏已有的 `super-admin`
- AI IDE 保持部署在 `8080`
- 阿里云安全组必须手动放行 `8080`
- `1G` 内存环境不适合在服务器上执行前端 `vite build`

## 3. 本次实际部署过程

### 3.1 初始目标

本次目标是将 AI IDE 部署到阿里云服务器，并确保：

- 不影响现有 `super-admin`
- 可以通过 `8080` 独立访问 AI IDE
- 保持前后端与数据库容器可运行

### 3.2 初始部署思路

最开始按常规 Docker 部署方式推进，核心思路包括：

1. 在服务器上准备项目目录和运行目录
2. 启动或重建数据库、后端、前端相关容器
3. 对前端执行构建，再通过 nginx 对外提供静态资源
4. 后端容器暴露 `8000`，前端通过 `8080` 暴露

### 3.3 中途调整

部署过程中发现服务器资源过低，前端在服务器端构建风险很高，因此后续策略调整为：

1. 前端在本地构建 `dist`
2. 将本地构建产物上传到服务器
3. 服务器仅执行轻量级 nginx 镜像构建
4. 避免在服务器上执行高内存占用的 `vite build`

这也是后续最终确认的推荐方案。

## 4. 过程中发生的主要问题

### 4.1 前端构建疑似卡死

#### 现象

部署时出现“安装依赖/构建像卡住一样”的情况。

#### 根因判断

并非真的逻辑卡死，核心原因更可能是：

- 服务器只有 `1G` 内存
- 前端 `vite build` 在低内存机器上容易触发内存不足
- 构建阶段可能出现长时间无输出、极慢，甚至 OOM 假死

#### 结论

不能继续采用“服务器现场构建前端”的方式。

### 4.2 误碰现有 `80` 端口服务风险

#### 现象

部署过程中一度引发用户担心原有地址被覆盖，说明部署行为已经接近或触碰到现有 `super-admin` 服务。

#### 影响

- 有可能影响现有线上系统
- 用户明确要求不要再改动 `80` 端口上的现有服务

#### 最终约束

- `super-admin` 固定保留在 `80`
- AI IDE 固定使用 `8080`
- 除非用户再次明确授权，否则不再尝试把 AI IDE 改成 `/ai/` 挂到 `80`

### 4.3 `8080` 外网无法访问

#### 现象

AI IDE 的访问地址 `http://8.130.118.128:8080` 无法打开。

#### 根因判断

最可能原因不是容器本身，而是阿里云安全组未放行 `8080` 入方向端口。

#### 处理建议

在阿里云控制台中增加安全组规则：

- 方向：入方向
- 协议：TCP
- 端口范围：`8080/8080`
- 授权对象：`0.0.0.0/0` 或按需限制来源 IP

### 4.4 SSH 连接不稳定 / 重启后失联

#### 现象

后续排障时出现 SSH 无法稳定连接、重启后仍无法继续正常检查的情况。

#### 根因判断

可能原因包括：

- `1G` 内存环境下容器启动后资源被迅速吃满
- 若配置了激进的自动重启策略，服务反复拉起会进一步争抢资源
- SSH 进程得不到足够资源，导致远程连接异常或超时

#### 经验教训

低内存服务器不要使用“服务端编译 + 自动无限重启”组合。

### 4.5 域名与路径方案讨论后被否决

#### 讨论过的方案

曾评估将 AI IDE 放到 `80` 端口下的 `/ai/` 路径，通过 nginx 网关转发。

#### 为什么没有继续

该方案虽然理论可行，但会引入额外复杂度：

- 前端需要调整 base path 到 `/ai/`
- nginx 转发规则更复杂
- 容易影响现有 `super-admin`
- 与用户“不要动 `80`”的要求冲突

#### 当前结论

该方案视为已放弃，除非用户后续重新提出。

## 5. 最终形成的可行部署策略

### 5.1 推荐方案

当前最适合该服务器配置的方式是：

1. 本地执行前端构建，产出 `dist`
2. 将 `dist`、`nginx.conf` 以及精简版 Dockerfile 上传到服务器
3. 在服务器上仅执行轻量级 nginx 镜像构建
4. 前端容器通过 `8080:80` 暴露
5. 后端沿用已构建镜像，仅重启容器
6. 启动后调用后端接口初始化内置 Agent

### 5.2 为什么这个方案更稳

优点如下：

- 避开服务器 `vite build` 的 OOM 风险
- 服务器只做静态文件 COPY，构建速度快
- 对 `1G` 内存 ECS 更友好
- 不需要改动 `super-admin` 的 `80` 端口

## 6. 已准备的辅助脚本

部署目录下已经存在一批辅助脚本，可继续复用：

- `deploy/deploy_frontend.py`
- `deploy/remote.py`
- `deploy/start_services.py`
- `deploy/start_services2.py`
- `deploy/deploy_ssh.py`
- `deploy/rebuild.py`

其中：

- `deploy_frontend.py` 的作用是将本地 `dist` 打包上传到服务器，并在远端构建轻量前端镜像
- `remote.py` 的作用是快速执行远程命令，例如 `free -h`、`docker ps` 等

## 7. 当前推荐的标准部署步骤

### 7.1 前端

在本地机器完成：

1. 安装前端依赖
2. 执行前端构建，生成 `dist`
3. 检查 `nginx.conf`
4. 使用上传脚本将 `dist` 上传到服务器

### 7.2 服务器侧

服务器上仅执行以下轻量动作：

1. 解压上传包
2. 以精简 Dockerfile 构建前端镜像
3. 重启 `ai-ide-frontend`
4. 重启 `ai-ide-backend`
5. 检查容器状态
6. 调用初始化接口

### 7.3 网络侧

额外必须确认：

1. 安全组已放行 `8080`
2. 容器端口映射正确
3. 服务器本机防火墙未拦截 `8080`

## 8. 后续排障清单

如果再次出现“页面打不开”或“部署后失联”，建议按以下顺序排查：

1. 检查阿里云安全组是否放行 `8080`
2. 检查 `docker ps` 是否能看到前后端容器
3. 检查前端容器是否映射 `8080:80`
4. 检查后端容器是否映射 `8000:8000`
5. 在服务器执行 `free -h` 查看剩余内存
6. 查看 `docker logs ai-ide-frontend`
7. 查看 `docker logs ai-ide-backend`
8. 若 SSH 无法连接，优先通过阿里云 VNC 进入系统排障

## 9. 风险与注意事项

### 9.1 资源风险

- `1G` 内存是当前部署最大风险点
- 不建议在服务器执行任何重型编译任务

### 9.2 端口风险

- 不要再动 `80`
- AI IDE 按既定方案放在 `8080`

### 9.3 凭证风险

- 之前部署过程中使用过服务器登录信息和模型 API Key
- 后续应统一迁移到 `.env` 或安全密钥管理方案中
- 文档、脚本、截图、聊天记录中都不应再明文传播敏感信息

## 10. 当前结论

本次部署复盘后，已经形成较明确的结论：

1. 该服务器不适合“前端在服务端现编译”的部署方式
2. AI IDE 应继续独立运行在 `8080`
3. `super-admin` 必须保留在 `80`
4. 阿里云安全组放行 `8080` 是外网访问前提
5. 最稳妥方案是“本地 build 前端 + 上传 dist + 远端轻量部署”

## 11. 建议的下一步

若后续继续部署，推荐按下面顺序执行：

1. 先确认阿里云安全组已放行 `8080`
2. 通过 SSH 或 VNC 检查当前服务器可用内存
3. 本地重新构建前端 `dist`
4. 使用轻量上传脚本重新部署前端
5. 重启后端容器
6. 验证 `http://服务器IP:8080`
7. 验证 `super-admin` 的 `80` 端口仍正常
